# TIP 3 ÔÇö `sticky` CLI

> **Phase goal:** ship the `sticky` CLI as a single-file native executable. The CLI is purely a thin wrapper over `src/core/` filesystem operations ÔÇö it does **not** talk to plasmashell. Once installed, `sticky <text>` writes a file; the daemon (TIP 6) and widget (TIPs 4ÔÇô5) make it appear on the desktop. Until those land, the CLI is fully testable against the on-disk directory.

## 1. Scope

In:

- `src/cli/main.ts` entry point.
- Argument parsing for the four documented commands.
- Help / version output.
- Stable, machine-readable `--json` output mode for `-l` (helps shell scripting and tests).
- `bun build --compile` producing `dist/sticky` (single static binary, ~50 MB).
- Integration tests that spawn the binary against a temp `XDG_DATA_HOME`.
- Man page stub (`tools/man/sticky.1`) generated or hand-written.
- Shell completions for bash + zsh (generated by parser if practical, otherwise hand-written).

Out:

- D-Bus to plasmashell (intentionally ÔÇö daemon owns that).
- `$EDITOR` invocation (CLI replaces content directly per spec).
- A subcommand to start/stop the daemon (TIP 6).

## 2. Command surface (final, locked from plan ┬º5)

```text
sticky <text>             Create a new note containing <text>. Prints the new id.
sticky -l, --list         List all notes (id and truncated first line). --json for machine output.
sticky -d, --delete <id>  Delete the note with <id>. Removes file; daemon will despawn the widget.
sticky -e, --edit <id> <text>
                          Replace contents of <id> with <text>. Widget refreshes via its file watcher.
sticky -h, --help         Show help.
sticky -V, --version      Show version (synced from package.json).
```

Behavioural rules:

- Exit code 0 on success, 1 on user error (unknown id, invalid args), 2 on internal error.
- `<text>` may be `-` to read from stdin (useful for `echo ÔÇª | sticky -`).
- All writes go through `src/core/` ÔÇö CLI contains zero filesystem code of its own.
- `sticky` with no args prints help and exits 1 (so a forgotten arg is an error, not an empty note).

## 3. Implementation notes

- **Argument parser.** Use a minimal hand-rolled parser or a tiny dep (`mri`, ~1 KB). Avoid `commander`/`yargs` (heavy, slow startup, awkward to compile).
- **Startup.** `bun build --compile --minify --target=bun-linux-x64 src/cli/main.ts --outfile dist/sticky`. Cold-start budget Ôëñ 50 ms on a 2026-class laptop.
- **Output.** Default human output for `-l`:

  ```text
  2026-04-20-133412-7f2a  Buy milk
  2026-04-19-091205-b3d0  Refactor positions storeÔÇª
  ```

  `--json` output for `-l`:

  ```json
  [{"id":"2026-04-20-133412-7f2a","firstLine":"Buy milk","mtime":"2026-04-20T13:34:12.000Z"}]
  ```

- **Error UX.** Unknown id ÔåÆ `sticky: no note with id 'xyz'` to stderr, exit 1.
- **Concurrency.** CLI is short-lived; no debounce concern. `createPositionsStore` is not used by the CLI at all ÔÇö only the widget writes positions.

## 4. Test plan

### Unit (bun test, in-process)

- `args.test.ts`: parse all flag forms (`-d X`, `--delete=X`, `--delete X`).
- `commands.test.ts`: each command invoked as a function (export `run(argv): Promise<number>`) returns the right exit code and writes the right files in a tmp `XDG_DATA_HOME`.

### Integration (bun test, spawning the compiled binary)

- Build `dist/sticky` once in `beforeAll`.
- Cases:
  - `sticky "hello"` ÔåÆ exit 0, file appears, stdout is the new id.
  - `sticky -l` after creating 3 notes ÔåÆ 3 lines, sorted by mtime desc.
  - `sticky -l --json` parses as JSON array of length 3.
  - `sticky -d <id>` ÔåÆ file is gone.
  - `sticky -d badid` ÔåÆ exit 1, error on stderr.
  - `sticky -e <id> "new content"` ÔåÆ file content equals "new content".
  - `echo "stdin note" | sticky -` ÔåÆ file content equals "stdin note".

### Property (fast-check)

- For any sequence of `create`/`delete`/`edit` ops on random ids/texts, `sticky -l --json` matches the in-memory expected state.

## 5. Implementation steps

1. Create `src/cli/{main,args,commands,output}.ts`. SPDX headers.
2. Implement `args.ts` (parser) + tests.
3. Implement `commands.ts` (one function per command, each takes a typed args object and returns `Promise<number>`) + tests.
4. Implement `main.ts` ÔÇö wires argv ÔåÆ args ÔåÆ commands ÔåÆ exit code.
5. Add `bun run build:cli` script: `bun build --compile --minify src/cli/main.ts --outfile dist/sticky`.
6. Add CI step that builds the binary and runs the integration suite against it. Cache the binary as a workflow artifact.
7. Hand-write `tools/man/sticky.1` (or generate from a tiny YAML descriptor).
8. Hand-write completions in `tools/completions/{sticky.bash,_sticky}`.

## 6. Acceptance criteria

- `bun run build:cli` produces `dist/sticky` Ôëñ 60 MB.
- All integration tests pass against the compiled binary.
- `dist/sticky --help` output matches the spec in ┬º2 verbatim.
- `dist/sticky --version` prints the `package.json` version.
- Cold-start (`time dist/sticky --version`) Ôëñ 50 ms on dev machine; recorded in CI as a non-failing benchmark line.

## 7. Done = ready for TIP 4

CLI is the first end-user-visible artefact. Even before the widget ships, a power user could `sticky "hello"` and inspect the file by hand. TIP 4 brings the desktop visible.
